# Hydrus Client (web)

Быстрый веб-клиент для [Hydrus Network](https://hydrusnetwork.github.io/hydrus/),
работает напрямую с Client API. SolidJS + Vite + TypeScript.

## Возможности

- **Поиск** по тегам с нечётким автокомплитом (как в самом Hydrus); пустой запрос =
  `system:everything`. Сортировки (`file_sort_type`), выбор тег-домена и файлового домена,
  трёхпозиционный NSFW-фильтр с настраиваемыми тегами.
- **Виртуализированная сетка** миниатюр без сторонних либ — десятки тысяч файлов скроллятся
  плавно; превью видео скрабится мышью, gif/apng анимируются.
- **Просмотр** (изображение/видео/Flash через Ruffle) с зумом/паном, навигацией с клавиатуры
  и сайдбаром тегов/рейтингов/заметок/URL.
- **Редактирование** тегов и рейтингов с оптимистичными записями и undo.
- **Массовые операции** над выделением: архив/инбокс, корзина и восстановление, теги,
  рейтинги, заметки, URL, отношения файлов, копирование хэшей/URL, экспорт в один ZIP.
- **Дубликаты** через перцептивные хэши Hydrus: авто-разрешение по компаратору и
  ручной фильтр по одной паре.
- **Импорт**: локальные файлы/папки прямо в Hydrus (работает и в статической сборке),
  а под dev-сервером ещё и через gallery-dl со стейджингом (см. ниже).
- **Состояние в URL-хэше** (запрос/сортировка/домены/открытый файл) — ссылки шарятся и
  восстанавливаются при перезагрузке. Тема, звук превью и настройки хранятся в `localStorage`.

## Подготовка Hydrus

1. В Hydrus: **services → manage services → client api** — включить, запомнить порт
   (по умолчанию `45869`).
2. Создать **access key**. Минимум — права на поиск и просмотр файлов; для записи нужны
   ещё Edit Tags / Edit Ratings / Manage Notes / Manage URLs / Manage File Relationships,
   для импорта — Import Files (+ Add Tags / Import URLs для gallery-dl).
3. Включить **support CORS headers** (браузерный клиент ходит с другого origin).

## Запуск

```bash
cp .env.example .env   # вписать адрес и ключ
npm install
npm run dev            # http://localhost:5173
```

Настройки задаются тремя способами (приоритет сверху вниз):
`localStorage` (форма в UI) → `.env` (`VITE_HYDRUS_URL`, `VITE_HYDRUS_KEY`) → дефолт.
Если в `.env` заданы оба значения — приложение стартует сразу в сетке, без формы.

> `VITE_*` вшиваются в бандл при сборке, ключ окажется в JS. Нормально для
> локального использования; **не публикуй такую сборку с реальным ключом.**

Сборка статики: `npm run build` → каталог `dist/` (отдаётся любым статик-сервером).


## Архитектура

- `src/api/hydrus.ts` — типизированный клиент API. JSON-запросы (поиск/метаданные/записи)
  идут через `fetch` с ключом в заголовке; URL миниатюр/файлов для `<img>/<video>` несут
  ключ в query-параметре (на медиа-теги заголовки не повесить). URL стабильны по `file_id`
  → агрессивное кеширование браузером (контент иммутабелен). Метаданные кешируются в два
  уровня: полные (с тегами) для вьюера и батчируемые базовые для сетки.
- `src/App.tsx` — центральный владелец состояния (сигналы SolidJS, без стора).
- `src/components/VirtualGrid.tsx` / `VirtualList.tsx` — оконная виртуализация без зависимостей.
- `src/components/FileViewer.tsx` — просмотр с оптимистичными правками и undo.
- gallery-dl-импорт — **только под `npm run dev`**: Vite-плагин в `vite.config.ts` запускает
  gallery-dl на хосте и отдаёт middleware `/__gallerydl`. В статической сборке этого нет,
  поэтому вкладка импорта показывает только локальные файлы.
